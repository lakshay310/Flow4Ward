## 📌 What is Flow4Ward?

**Flow4Ward** is a real-time traffic intelligence command center built for Bengaluru traffic authorities. It combines:

- 🤖 **Machine Learning** — 3 scikit-learn models trained on 8,205 real Astram incidents to predict traffic impact, road closures, and officer requirements
- 📡 **Live WebSocket Traffic** — Real-time congestion data streamed every 30 seconds across 6 Bengaluru zones
- 🗺️ **Junction Impact Mapping** — Risk analysis for 10 critical Bengaluru junctions (Silk Board, Hebbal, KR Puram, etc.)
- 👮 **Resource Allocation Optimizer** — Automated officer & barricade deployment planning
- 📊 **Historical SOPs** — Data-driven standard operating procedures derived from real incident patterns

## ⚡ Quick Start (Local Development)

```bash
# 1. Clone the repo and enter the directory
cd "hackathon round2"

# 2. Install all dependencies
npm run install:all

# 3. Configure environment
cp backend/.env.example backend/.env   # then paste your Neon DATABASE_URL

# 4. Initialize the database schema (run once)
node backend/utils/initDB.js

# 5. Seed with Bengaluru sample data
npm run seed

# 6. Start both servers
npm run dev
```

Then open: **http://localhost:5173**

> **Note:** `npm run dev` automatically kills any existing processes on ports 5000 & 5173, starts both servers, and opens the browser.

## 🌟 Key Features

### 1. 🤖 AI Decision Simulator
- Shows predicted risk **before** vs **after** applying intervention
- One-click "Apply AI Plan" — deploys officers, places barricades, activates diversion
- Calculates % risk reduction (e.g. 77 → 53 = **31% reduction**)
- Powered by `mlPlaceholder.js` reading weights trained on the real Astram dataset

### 2. ⏱️ Event Impact Timeline
- Hour-by-hour risk bars from pre-event setup through post-event dispersal
- Color-coded severity: Minimal → Low → Moderate → High → Severe → Critical
- Identifies the peak congestion window automatically

### 3. 🗺️ Junction Impact Prediction
- Analyzes **10 real Bengaluru junctions** (Silk Board, Hebbal, KR Puram, Marathahalli, etc.)
- Risk score = proximity-weighted model output + data-derived baseline per junction
- Surfaces critical junction warnings with immediate deployment alerts

### 4. 👮 Resource Allocation Optimizer
- Auto-assigns officer counts per junction based on weighted risk
- Shows officer ranges, barricade count, and priority tier per deployment zone
- Total officer count derived from the ML officer-count regression model (MAE ≈ 1.6 officers)

### 5. 🎯 Live Traffic Control Room
- Real-time congestion map (6 Bengaluru zones) updated every 30 seconds via Socket.io + cron
- Congestion trend chart (last 6 hours via Recharts)
- Live alert feed + upcoming events panel
- IT corridor aware — East & Central zones get a realistic 8-point congestion boost

### 6. 📊 Historical Insights
- Data-driven SOPs per incident type (vehicle breakdown, accident, water logging, public event, etc.)
- Derived from 8,205 real Astram incidents: resolution percentiles, corridor risk multipliers, rush-hour patterns

## 🤖 ML Pipeline (`ml/train.py`)

Trains **3 scikit-learn models** on the Astram Bengaluru incident dataset and exports a `model_weights.json` used at runtime by the Node.js backend:

| Model | Type | Metric |
|---|---|---|
| **Impact Score Regressor** | RandomForest (200 trees, depth 10) | MAE ≈ 8.4 pts, R² ≈ 0.51 |
| **Road Closure Classifier** | GradientBoosting (100 estimators) | Accuracy ≈ see output |
| **Officer Count Regressor** | GradientBoosting (100 estimators) | MAE ≈ 1.6 officers |

**Features used:** `requires_road_closure`, `is_planned`, `is_high_priority`, `is_rush_hour`, `is_weekend`, `is_orr`, `is_cbd`, `is_non_corridor`, `hour_of_day`, `day_of_week`, `cause_enc`, `corridor_enc`, `zone_enc`

**What gets exported to `model_weights.json`:**
- `causeWeights` — impact weight per incident cause
- `corridorRisk` — risk multiplier per corridor (ORR, CBD, Tumkur Rd, etc.)
- `zoneBaseline` — congestion baseline per zone (normalized to 45–80)
- `rushHourMultipliers` — per-hour congestion multiplier
- `junctions` — 10 key junctions with data-derived incident-density baselines
- `corridorJunctions` — corridor → junction mapping for impact propagation
- `historicalInsights` — SOP recommendations per incident type
- `officerEstimates`, `causeStats`, `featureImportances`, `constants`

### Run the ML training

```bash
pip install pandas numpy scikit-learn joblib
python ml/train.py
```

The script reads `Astram event data_anonymized - ...csv` from the project root and writes `backend/utils/model_weights.json`. Restart the backend to pick up the new weights.

## 🔌 API Endpoints

### Events
| Method | Route | Description |
|---|---|---|
| `GET` | `/api/events` | List all events (filter: `type`, `status`, `severity`, `zone`) |
| `POST` | `/api/events` | Register new event → auto-generates prediction + alert |
| `GET` | `/api/events/:id` | Single event detail |
| `GET` | `/api/events/stats` | Counts: total, upcoming, active, critical |
| `PATCH` | `/api/events/:id` | Update event fields |
| `DELETE` | `/api/events/:id` | Remove event |

### Predictions
| Method | Route | Description |
|---|---|---|
| `GET` | `/api/predictions` | All predictions (joined with event metadata) |
| `GET` | `/api/predictions/event/:eventId` | Latest prediction for an event |
| `POST` | `/api/predictions/generate` | Re-run ML prediction for `{ eventId }` |
| `POST` | `/api/predictions/simulate` | Run intervention simulation (officers, barricades, diversion) |
| `PATCH` | `/api/predictions/:id/resources` | Override resource allocation |

### Traffic
| Method | Route | Description |
|---|---|---|
| `GET` | `/api/traffic/live` | Latest reading per zone |
| `GET` | `/api/traffic/summary` | Aggregated zone summary (avg congestion, label) |

### Alerts
| Method | Route | Description |
|---|---|---|
| `GET` | `/api/alerts` | All alerts (filter: `severity`, `resolved`) |
| `GET` | `/api/alerts/stats` | Counts by severity |
| `PATCH` | `/api/alerts/:id/resolve` | Mark alert as resolved |

### Health
| Method | Route | Description |
|---|---|---|
| `GET` | `/api/health` | Server status + timestamp |
